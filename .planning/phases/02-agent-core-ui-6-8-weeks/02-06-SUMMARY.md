# Phase 02 Plan 06: Keycloak v26 + uv Migration + PKCE Auth Flow Summary

**One-liner:** Migrated to uv/pyproject.toml, configured Keycloak v26 pointing to oauth2.qa.comsatel.com.pe, implemented PKCE Authorization Code Flow in frontend auth store.

### Task 2 — Keycloak config + docker-compose (4aca1b2)

- Updated `.env.example` with Keycloak QA host values

### Task 3 (scope change) — PKCE public client (2441915, 10237b8)
The original plan called for a confidential client with client secret. After user confirmed the Keycloak client is **public** with Standard Flow + PKCE enabled, the implementation was changed:

**Backend (2441915):**
- Removed `KEYCLOAK_CLIENT_SECRET` from `config.py` (public client needs none)
- Added `KEYCLOAK_JWKS_URL` property for JWT verification via JWKS endpoint
- Updated `.env.example`: removed client secret, added `VITE_KEYCLOAK_*` frontend vars with PKCE explanation comment

**Frontend (10237b8):**
- Rewrote `src/frontend/src/stores/auth.ts` with full PKCE implementation:
  - `generateCodeVerifier()` — cryptographically random 32-byte verifier
  - `generateCodeChallenge()` — SHA-256 hash encoded as base64url
  - `login()` — stores verifier in sessionStorage, redirects to Keycloak auth endpoint
  - `handleCallback(code)` — exchanges auth code + verifier for tokens
  - `setTokens()` — stores access/refresh/id tokens in localStorage under `mia_*` keys
  - `logout()` — clears tokens, redirects to Keycloak logout endpoint with `id_token_hint`
- Updated `App.vue`: on mount, detect `?code=` in URL, call `handleCallback`, clean URL, navigate to `/agents`
- Updated `router/index.ts`: `beforeEach` guard redirects unauthenticated users to `auth.login()`; `/` is public to allow callback landing

## Key Decisions

| Decision | Rationale |
|---|---|
| PKCE instead of confidential client | Keycloak client is configured as public — no secret is available or appropriate for a browser SPA |
| Redirect URI is root `/` | Avoids adding a dedicated `/callback` route; App.vue handles detection on mount |
| `sessionStorage` for verifier | Survives the redirect but is tab-isolated, appropriate for PKCE security model |
| `localStorage` for tokens | Consistent with existing pattern; httpOnly cookies planned for later phase |
| No separate auth page/route | Login is triggered imperatively via `auth.login()` which redirects to Keycloak |

## Deviations from Plan

### Scope Change (user-approved) — Task 3 architecture change

**Found during:** Task 3 checkpoint review

**Original plan:** Confidential client with `KEYCLOAK_CLIENT_SECRET`, backend token validation

**Change:** Public client with PKCE Authorization Code Flow, browser-side token exchange

**Why:** User confirmed Keycloak client is configured as public (no Credentials tab / no client secret). A public client with PKCE is the correct approach for a browser-based SPA. Using a confidential client secret in frontend code would be a security anti-pattern.

**Impact:** Removed `KEYCLOAK_CLIENT_SECRET` from config and `.env.example`. Added PKCE utilities, full auth store, App.vue callback handler, and router guard. Scope expanded significantly but all within browser-only code.

## Known Stubs

- `auth.ts` `user` state is never populated from the decoded JWT — `user` is always `null`. A future plan should decode the access token (or call the userinfo endpoint) and populate this field.
- Router guard calls `auth.login()` asynchronously but does not call `next(false)` — in Vue Router 4 with async actions this is acceptable (navigation is interrupted by `window.location.href`), but the missing `next(false)` is technically incomplete.
