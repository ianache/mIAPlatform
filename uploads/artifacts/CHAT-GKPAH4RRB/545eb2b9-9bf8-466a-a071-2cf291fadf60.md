## 3️⃣ Observaciones y recomendaciones  

1. **Incluir datos de prueba concretos**  
   - Añadir un ejemplo de **Orden de Trabajo** (n.º, fecha) y del **ítem** (ID, descripción, VIN). Esto facilita la reproducción y la posible automatización futura.  

2. **Especificar entorno de ejecución**  
   - Versión exacta de la app (p. ej. 6.3.2).  
   - Sistema operativo y versión del dispositivo (Android 13, iOS 17).  
   - Si el test se ejecuta en modo “online” (con backend real) o “mock”.  

3. **Definir post‑condiciones**  
   - Qué estado debe quedar el ítem después de la verificación exitosa (p. ej. “Verificado”).  
   - Si el test debe restaurar el ítem a “Por verificar” para permitir re‑ejecución, indicar cómo hacerlo (p. ej. “Reabrir ítem mediante herramienta de administración”).  

4. **Criterio de falla explícito**  
   - “Falla si la app envía más de una petición HTTP al endpoint `/telemetry/verify`”.  
   - “Falla si la acción vuelve a estar habilitada antes de recibir la respuesta”.  

5. **Considerar casos negativos opcionales** (no obligatorios para este CP, pero útiles para la suite):  
   - Falta de conectividad → mensaje de error.  
   - Permisos GPS deshabilitados → mensaje de advertencia.  

6. **Revisión de la tabla “Given/When/Then”**  
   - El “Then” solo menciona que la app inicia la verificación y muestra “Verificando…”. Sería conveniente añadir “y deshabilita la acción”.  

7. **Formato de la tabla de criterios de aceptación**  
   - Se podría alinear con la convención de BDD:  

     ```
     Given   El técnico está en la OT con ítem en estado "Por verificar"
     When    Presiona la acción "Por verificar"
     Then    La app muestra "Verificando…" y deshabilita la acción
     And     No se generan peticiones duplicadas
     ```
