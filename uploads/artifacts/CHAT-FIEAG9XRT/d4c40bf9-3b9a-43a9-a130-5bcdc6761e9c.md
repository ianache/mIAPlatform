## 4. Observaciones y Recomendaciones  

1. **Verificar el estado final**  
   - Añadir al final del procedimiento un paso que confirme que, una vez recibida la respuesta del servicio, la UI muestra el estado correspondiente (por ejemplo, “Verificado” o “Error”).  
   - Esto garantiza que el bloqueo se libera correctamente y que el flujo completo se cierra.

2. **Manejo de errores**  
   - Si la US contempla escenarios de falla (pérdida de conectividad, timeout, respuesta de error del backend), incluir al menos un caso negativo que compruebe que la acción vuelve a estar habilitada y muestra mensaje de error al usuario.  
   - Si no corresponde a este CP, crear un CP adicional (p.ej., US5430‑CP02).

3. **Medición de solicitudes duplicadas**  
   - Definir cómo se valida “no se generan solicitudes duplicadas”. Por ejemplo, mediante captura de logs del cliente, inspección de la cola de peticiones en el backend, o uso de una herramienta de proxy (Charles, Wireshark).  
   - Añadir ese detalle al paso 8 para que el tester tenga una forma concreta de comprobarlo.

4. **Datos de prueba opcionales**  
   - Documentar que el ítem debe estar configurado con telemetría válida (p.ej., número de VIN, IMEI) para que la llamada al backend sea aceptada.  
   - Esto ayuda a crear entornos de prueba reproducibles.

5. **Limpieza posterior**  
   - Incluir un paso de “Post‑condición” que indique que el ítem vuelve a su estado original (si se está ejecutando en un entorno de pruebas) o que se elimina cualquier dato temporal creado.
